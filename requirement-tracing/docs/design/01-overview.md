# 需求追溯系统 - 设计概览

## 1. 项目背景

### 1.1 目标
构建需求文档（PDF/Excel）与代码库（Test Case）的对齐分析系统，实现：
- **可追溯性验证**（优先级1）：查询需求被哪些Test Case覆盖
- **代码追溯查询**（优先级2）：查询Test Case对应哪些需求
- **差距分析报告**（优先级3）：生成需求覆盖率报告
- **影响分析**（优先级4）：分析需求变更的影响范围
- **版本对比**（优先级5）：对比不同版本需求文档的覆盖率变化

### 1.2 项目规模
- 需求文档：10-50个（Excel + PDF）
- Test Case：100-1000个（Python文件）
- 代码文件：500-5000个

### 1.3 核心挑战
1. **需求文档格式不统一**：Excel结构不固定，PDF为自由文本
2. **Test Case无显式需求ID**：需要通过语义关联建立联系
3. **关联准确性要求高**：需要多阶段策略平衡召回率和准确率
4. **LLM深度参与预处理**：需要从非结构化文档中提取结构化需求

## 2. 技术选型

### 2.1 核心技术栈
- **文档解析**：MinerU（PDF/Excel → Markdown）
- **需求提取**：本地LLM（Ollama + Qwen2.5:14b）
- **索引和检索**：LlamaIndex（向量索引 + BM25）
- **代码解析**：AST解析 + LLM摘要生成

### 2.2 架构选择
**方案A：分层架构 + 离线索引**（已选定）

```
┌─────────────────────────────────────────────────────┐
│                   应用层 (CLI/API)                   │
└────────────────────┬────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────┐
│                  工作流层 (Workflows)                │
│  - 需求覆盖率查询  - 代码追溯查询                    │
│  - 差距分析报告    - 版本对比                        │
└────────────────────┬────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────┐
│                  匹配层 (Matching)                   │
│  - 多阶段检索器 (BM25 → 向量 → LLM)                  │
│  - 置信度评估   - 可解释性生成                       │
└────────────────────┬────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────┐
│                  索引层 (Indexing)                   │
│  - 需求索引 (Vector + BM25)                          │
│  - Test Case索引 (Vector + BM25)                     │
│  - 版本管理                                          │
└────────────────────┬────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────┐
│                 预处理层 (Preprocessing)             │
│  - 文档解析 (MinerU)                                 │
│  - 需求提取 (LLM)                                    │
│  - Test Case提取 (AST + LLM)                         │
└─────────────────────────────────────────────────────┘
```

**优点**：
- 预处理和查询分离，查询速度快
- 索引持久化，支持版本管理
- 架构清晰，易于扩展和维护

## 3. 数据模型

### 3.1 需求 (Requirement)
```python
{
    "title": str,           # 需求标题/摘要
    "description": str,     # 详细描述
    "type": str,            # 需求类型（功能/性能/兼容性等）
    "keywords": List[str],  # 关键词/标签
    "source": {
        "file": str,        # 源文件名
        "version": str,     # 版本号（从文件名提取）
        "chapter": str,     # 章节信息
        "page": int         # 页码（PDF）
    }
}
```

### 3.2 Test Case
```python
{
    "file_path": str,       # 文件路径
    "file_name": str,       # 文件名
    "platform": str,        # Windows/Linux
    "category": str,        # Common/Customer
    "subcategory": str,     # Performance/Compatibility等
    "summary": str,         # 摘要（docstring或LLM生成）
    "keywords": List[str],  # 关键词
    "functions": List[str]  # 函数列表
}
```

### 3.3 匹配结果 (MatchResult)
```python
{
    "doc_id": str,          # 文档ID
    "text": str,            # 文本内容
    "metadata": dict,       # 元数据
    "score": float,         # 匹配分数 (0-1)
    "confidence": str,      # 置信度 (high/medium/low)
    "match_reason": str     # 匹配原因说明
}
```

## 4. 核心流程

### 4.1 索引构建流程
```
需求文档 → MinerU解析 → LLM提取结构化需求 → 构建索引
Test Case → AST解析 → LLM生成摘要 → 构建索引
```

### 4.2 查询流程（三阶段检索）
```
查询 → 阶段1: BM25快速召回 (top 50)
     → 阶段2: 向量精确匹配 + 混合评分 (top 20)
     → 阶段3: LLM精确判断 (top 5, 可选)
     → 返回结果 + 置信度 + 可解释性
```

### 4.3 版本管理流程
```
新版本文档 → 提取版本号 → 构建独立索引
版本对比 → 识别新增/删除/修改需求 → 对比覆盖率变化
```

## 5. 关键设计决策

### 5.1 为什么选择离线索引？
- 需求文档和Test Case变化频率低
- 查询频率相对较高
- 离线索引可以提供更快的查询响应

### 5.2 为什么使用三阶段检索？
- **阶段1 (BM25)**：快速召回，保证召回率
- **阶段2 (向量)**：语义匹配，提高准确率
- **阶段3 (LLM)**：精确判断，用于高置信度场景

### 5.3 为什么需要置信度分级？
- 避免误报：低置信度结果需要人工审核
- 支持不同场景：差距分析可以只看高置信度结果
- 提供可解释性：帮助用户理解匹配原因

### 5.4 为什么Test Case使用AST + LLM混合方式？
- **优先使用docstring**：如果存在，直接使用（准确且快速）
- **无docstring时用LLM**：生成摘要（保证覆盖率）
- **AST提取元数据**：函数列表、路径信息（用于过滤和加权）

## 6. 验证策略

### 6.1 准确性目标
- **Precision > 0.8**：高置信度结果中，80%以上是正确的
- **Recall > 0.6**：实际相关的Test Case中，60%以上被召回

### 6.2 验证方法
1. **人工标注**：标注部分需求-Test Case对（100-200对）
2. **利用现有关联**：如果Test Case中有需求ID注释，作为ground truth
3. **迭代优化**：根据验证结果调整权重和阈值

## 7. 部署方式

### 7.1 本地CLI（推荐用于个人/小团队）
- 按需查询，数据完全本地
- 部署简单，无需服务器

### 7.2 Web服务（推荐用于团队协作）
- 支持多用户并发
- 提供Web UI界面
- 可以集成到CI/CD流程

### 7.3 定期报告（推荐用于自动化场景）
- 使用Cron定期生成差距分析报告
- 自动发送邮件通知

## 8. 扩展性考虑

### 8.1 支持更多文档格式
- 预处理层设计为插件式，可以添加新的Parser

### 8.2 支持更多代码语言
- Test Case提取器可以扩展支持Java、C++等

### 8.3 支持更多匹配策略
- 匹配层可以添加新的检索器（如Graph-based）

### 8.4 支持人工标注反馈
- 可以收集用户反馈，用于模型微调或规则优化
